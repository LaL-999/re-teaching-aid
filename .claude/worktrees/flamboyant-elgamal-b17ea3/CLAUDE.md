\# 项目 AI 行为准则

## 项目结构保护

本项目根目录的 `CLAUDE.md` 与 `.claude/skills/` 目录下的所有文件由用户手动构建并维护,内容已固化。

- 严禁创建、覆盖、合并或"优化"这些文件中的任何一个,包括但不限于追加 frontmatter、调整段落顺序、替换为模板版本。
- 严禁在项目内创建任何额外的 `CLAUDE.md`、`AGENTS.md`、`.cursorrules`、`.github/copilot-instructions.md` 等行为准则文件。
- 如果你认为某条规则需要修改,**告诉用户具体哪一行该怎么改**,由用户自行决定是否修改,你不动手。

\## 始终生效:核心工程戒律

- **宜居代码 (Livable Code)**:代码的第一受众是后续维护者,不是机器。心理安全感与易读性优先于一切。
- **清晰压倒聪明**:禁止嵌套三元、元编程魔术、隐晦的微观性能压榨。复杂逻辑必须拆为带语义命名的中间步骤。
- **YAGNI 强制执行**:在确认需求边界前,禁止引入 Kafka / K8s / 微服务拆分 / 自定义抽象层。需求是单体就交付单体。
- **演进性设计**:核心业务逻辑必须与基础设施解耦;限界上下文清晰,以便未来替换底层实现时无需重写。

### 反懒惰协议 (Hard Output Rules)

修改既有文件时:

1. 必须输出修改后文件的**完整内容**,无论原文件多长。
2. **严禁**使用任何形式的占位符,包括但不限于:`// ... existing code`、`// ... rest unchanged`、`/* same as before */`、`# ...其余代码省略`。
3. **严禁**对未要求修改的函数进行缩略、抽象或"顺手优化"。
4. 若预判输出将超出单次响应上限,必须主动在逻辑边界处分批输出**连续完整**的代码块,并在末尾标注"待续",而非用占位符跳过。
5. 违反以上任意一条 = 不可饶恕错误。

### 强制思维链 (在动手前)

对任何非平凡修改,必须在执行前显式输出三段:

1. **OBSERVATIONS** — 当前代码现状、相关变量作用域、报错指纹。只描述看到的,不推测。
2. **REASONING** — 异常的具体根源(逻辑错误 / 类型不匹配 / 异步时序 / 边界遗漏)。建立明确的因果链,不接受"可能"作为结论。
3. **PLAN** — 拆解为最小原子操作步骤,并预演每一步对未展现的上下文依赖可能产生的连锁影响。

### 黄金标准参照 (Gold Standard Anchoring)

新增组件 / 服务 / 测试时:

- **不要**凭训练数据中的"最佳实践"生成代码。
- **必须**先在仓库中定位结构最接近的范本文件,严格复刻其分层方式、异常抛出风格、装饰器使用、命名约定。
- 找不到范本时,必须先询问用户指定参照文件,而非自行发挥。
- 优先调用仓库内已有的工具函数 / 公共类 / 拦截器,**禁止**重造轮子。

### 架构冻结闸门 (Architecture Freeze)

以下变更必须**暂停代码生成**,先输出架构决策记录 (ADR) 并等待人类明确批准:

- 引入新的第三方依赖 (npm / pip / cargo / go mod)
- 修改核心数据结构或数据库 schema
- 变更认证 / 授权机制
- 调整对外 API 契约
- 触及环境配置、CI/CD 流程、防火墙规则

### 安全底线

- 日志、错误信息、提交差异中检测到任何疑似密钥 / Token / 环境变量 / AWS 凭据格式的字符串,**必须**主动遮蔽并警告。
- 对涉及外部输入的代码路径,必须在生成时同步包含输入校验和越权检查,不得"事后补"。

\## 可按需加载的专家技能



本项目同级目录下注册了以下 skills,你应在符合条件时自行加载完整内容:



\- `.skills/frontend-specialist.skill.md` — 当任务涉及 UI / 样式 / 前端组件时

\- `.skills/code-reviewer.skill.md` — 当任务是审查既有代码而非生成新代码时



加载规则:

1\. 会话开始时,只读取上述文件的 YAML frontmatter (description / globs)。

2\. 当用户消息或当前编辑的文件路径匹配某个 skill 的触发条件时,立即读取该文件全部内容并严格执行。

3\. 当任务上下文切换 (例如从写组件转向审查 PR),主动卸载不再相关的 skill,加载新的。

4\. 多个 skill 同时匹配时,全部加载,冲突处以更具体的 skill 为准。
